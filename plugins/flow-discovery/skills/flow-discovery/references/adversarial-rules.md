# Adversarial Review Rules

กฎหลักที่ทุก mode และทุก subagent ต้องปฏิบัติตาม (ดัดแปลงจาก BMAD Adversarial Review)

---

## RULE 1: ห้ามตอบ "ดีแล้ว" หรือ "ครบแล้ว"

**กฎ**: ต้องหา finding อย่างน้อย 3 ข้อเสมอ
**เหตุผล**: ไม่มีระบบใดที่สมบูรณ์แบบ ถ้าหาไม่ได้แปลว่ายังวิเคราะห์ไม่ลึกพอ
**วิธีปฏิบัติ**:
- ถ้าหา finding ได้น้อยกว่า 3 ข้อ → กลับไป analyze ใหม่ด้วยมุมมองอื่น
- ลองคิดจากมุม: ถ้าระบบนี้มี users 10,000 คน จะเกิดอะไรขึ้น?
- ลองคิดจากมุม: ถ้าระบบนี้ถูกโจมตี จะเจาะจุดไหนก่อน?

## RULE 2: ทุก finding ต้อง actionable

**กฎ**: ทุก finding ต้องระบุครบ 4 ข้อ:
1. **สถานการณ์** — อะไรอาจเกิดขึ้น
2. **Trigger** — เกิดเมื่อไหร่ / ทำอย่างไรถึงเกิด
3. **ผลกระทบ** — กระทบอะไร / ใครได้รับผลกระทบ
4. **แนะนำ** — วิธีแก้/ป้องกัน (2-3 ทาง)

**ตัวอย่าง ❌ ไม่ actionable**:
- "ระบบอาจมีปัญหา concurrency" → ไม่ระบุว่าตรงไหน ทำอย่างไรถึงเกิด

**ตัวอย่าง ✅ actionable**:
- "สถานการณ์: 2 คนแก้ไขข้อมูลสินค้าเดียวกันพร้อมกัน → Trigger: Admin A เปิดหน้าแก้ไข, Admin B เปิดหน้าเดียวกัน, ทั้งคู่กด Save → ผลกระทบ: ข้อมูลของคนที่ Save ทีหลังจะทับคนแรก → แนะนำ: (1) เพิ่ม optimistic locking ด้วย version field (2) แจ้งเตือนเมื่อข้อมูลถูกแก้ไขระหว่างที่กำลัง edit"

## RULE 3: ห้ามรายงาน false positive โดยเจตนา

**กฎ**: แยกชัดเจนระหว่าง "ปัญหาจริง" กับ "จุดที่ควรพิจารณา"
**วิธีปฏิบัติ**:
- **ปัญหาจริง** (Confirmed) — มีหลักฐานจาก code หรือ logic ชัดเจน
- **จุดที่ควรพิจารณา** (Needs Verification) — ไม่แน่ใจ 100% ต้องตรวจสอบเพิ่ม
- **ห้าม** สร้าง finding เพื่อให้ครบจำนวน ถ้าไม่พบจริง

## RULE 4: มองจากหลายมุม (Multi-Perspective)

**กฎ**: ทุก finding ต้องพิจารณาจากอย่างน้อย 3 มุมมอง:

| มุมมอง | คำถามหลัก |
|--------|----------|
| **User** | ใช้งานง่ายไหม? สับสนตรงไหน? ผิดพลาดง่ายตรงไหน? |
| **Developer** | maintain ง่ายไหม? มี tech debt ตรงไหน? test ครบไหม? |
| **Security** | มีช่องโหว่ตรงไหน? data leak ได้ไหม? auth bypass ได้ไหม? |
| **Business** | ตรง business rule ครบไหม? มี edge case ทาง business ไหม? |
| **Operations** | monitor/debug ได้ไหม? log ครบไหม? recover ได้ไหม? |

## วิธีใช้ใน Subagent Prompt

ทุก subagent prompt ต้องมี:
```
คุณต้องปฏิบัติตาม Adversarial Rules:
1. ห้ามตอบ "ดีแล้ว" — ต้องหา finding อย่างน้อย 3 ข้อ
2. ทุก finding ต้อง actionable — ระบุ สถานการณ์ + Trigger + ผลกระทบ + แนะนำ
3. ห้าม false positive — แยก "ปัญหาจริง" กับ "ควรพิจารณา"
4. มองหลายมุม — User, Developer, Security, Business, Operations
```
