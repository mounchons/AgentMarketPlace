# Monitor Plugin — Design Spec

## Overview

**Plugin name**: `monitor`
**Purpose**: Two-way interactive dashboard for visualizing and editing all plugin tracking files (*list.json), with live refresh and Claude Code task integration.

**Core loops**:
1. UI → JSON: User edits data in browser → saves back to JSON files on disk
2. JSON → UI: Claude Code modifies JSON → browser auto-refreshes via SSE
3. UI → Claude Code: User queues tasks from UI → Claude Code picks up and executes

## Architecture

```
Browser (localhost:3400)
    │
    │  REST API (GET/PUT)
    │  SSE stream (file-changed events)
    │
    ▼
monitor-server.js (Node.js, built-in http + chokidar)
    │
    │  read/write
    │  file watch
    │
    ▼
*list.json files + activity logs + task-queue.json
    │
    │  file changes detected
    │
    ▼
Claude Code (reads/writes JSON via plugins)
```

## Server Architecture

**Runtime**: Node.js with built-in `http` module. Single external dependency: `chokidar` for file watching.

**Port**: 3400 (configurable in config.json)

### Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | / | Serve index.html |
| GET | /*.js, *.css | Serve static GUI files |
| GET | /api/data | Read all available *list.json + activity logs |
| GET | /api/data/:source | Read single source (design/mockup/feature/qa) |
| PUT | /api/data/:source | Write back modified JSON to disk |
| GET | /api/events | SSE stream for live updates |
| POST | /api/task | Add task to task-queue.json |
| GET | /api/tasks | Read current task queue |

### File Watcher

- Uses chokidar to watch all paths defined in config.json
- On file change: SSE push `{type:"file-changed", source, timestamp}`
- Debounce: 500ms to prevent rapid-fire events
- Heartbeat: every 30s to keep SSE connection alive

### config.json

Auto-generated by `/monitor-init`. Only includes files that actually exist (graceful partial).

```json
{
  "port": 3400,
  "paths": {
    "design": "../design_doc_list.json",
    "mockup": "../.mockups/mockup_list.json",
    "feature": "../feature_list.json",
    "qa": "../qa-tracker.json"
  },
  "activity_logs": [
    "../.brain/activity-log.json",
    "../.bigbrain/activity-log.json",
    "../.agent/progress.md"
  ],
  "auto_open": true
}
```

## GUI Changes from tracker-gui

### FileLoader: Mode D (API)

Auto-detect: if URL is `localhost:PORT` → use Mode D (API fetch), else use Mode A (file drag-drop).

Mode D behavior:
- On page load: `GET /api/data` → load all available sources
- Skip landing screen → go directly to graph view
- Save: `PUT /api/data/:source` instead of browser download
- SSE: `EventSource('/api/events')` → auto-refresh on file changes

### New UI: Activity Timeline

In sidebar, replaces Legend section with collapsible timeline reading from activity logs:

```
ACTIVITY
├── 14:30 brain-save "User entity specs"
├── 14:25 feature #3 → passed
├── 14:20 qa-create-scenario --auto
└── [View all...]
```

### New UI: "Send to Claude" Button

In Detail Panel, dropdown button with options:
- **Copy**: copies command string (e.g. `/continue #5`) to clipboard + toast notification
- **Queue: implement**: writes task to task-queue.json with type "implement"
- **Queue: review**: writes task to task-queue.json with type "review"

## Commands

### `/monitor-init`

No arguments. Steps:
1. Scan project root for *list.json files
2. Scan for activity log files (.brain/, .bigbrain/, .agent/)
3. Create `.monitor/` directory
4. Copy template files from plugin templates/
5. Generate config.json with detected paths
6. Create empty task-queue.json
7. Run `npm install` in .monitor/
8. Add `.monitor/` to .gitignore
9. Display summary: "Found: design+ mockup+ feature+ qa-"

### `/monitor`

Optional argument: `--port=XXXX`

Steps:
1. Check `.monitor/` exists — if not, suggest `/monitor-init`
2. Check `node_modules/` exists — if not, run `npm install`
3. Start server: `node .monitor/monitor-server.js`
4. Auto-open browser: `start http://localhost:3400`
5. Display: "Monitor running at http://localhost:3400"

### `/monitor-task`

No arguments. Steps:
1. Read `.monitor/task-queue.json`
2. Display pending tasks for user to choose
3. Execute chosen task based on type:
   - `implement` → invoke /continue workflow
   - `review` → invoke /review workflow
   - `create-scenario` → invoke /qa-create-scenario
   - `create-mockup` → invoke /create-mockup
   - `edit-section` → invoke /edit-section
4. Mark task completed in queue
5. Server detects queue change → SSE → browser updates

## Hook: SessionStart

```json
{
  "hooks": [{
    "type": "prompt",
    "event": "SessionStart",
    "matcher": "startup|resume",
    "prompt": "Check if .monitor/ exists. If yes, read task-queue.json and show pending tasks count."
  }]
}
```

Notification only — does not auto-start server.

## Task Queue Schema

File: `.monitor/task-queue.json`

```json
{
  "tasks": [
    {
      "id": "task-001",
      "created_at": "2026-04-14T18:30:00Z",
      "type": "implement",
      "target": "feature:5",
      "label": "GET /api/users - List all",
      "source_node": "feature:5",
      "command_hint": "/continue #5",
      "status": "pending",
      "completed_at": null
    }
  ]
}
```

## Technical Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Server runtime | Node.js | SSE + file watch + REST API ecosystem; likely pre-installed |
| HTTP framework | Built-in http module | No Express needed for ~6 endpoints |
| File watcher | chokidar | Battle-tested, cross-platform, single dependency |
| Live updates | SSE (Server-Sent Events) | Simpler than WebSocket, one-direction is sufficient |
| Send to Claude | Clipboard + Task Queue | Clipboard for simplicity, queue for automation |
| Partial data | Graceful skip | Show what exists, empty state for missing sources |
| Browser open | Auto-open | Minimum friction |
| Deployment | .monitor/ in .gitignore | Per-developer, not committed |
